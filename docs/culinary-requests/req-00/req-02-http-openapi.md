# REQ-02 ‚Äî HTTP Client + OpenAPI-typed API layer

**Requested by:** Culinary frontend
**Priority:** P1 (needed once auth screen exists; workaround with `fetch` is acceptable for first week)
**Estimated effort:** small (2-4 days)

## Purpose

Provide a typed HTTP client that:
- Accepts an OpenAPI 3.1 schema (generated by FastAPI) and produces typed methods.
- Integrates with UIDetox reactivity ‚Äî a request returns a reactive resource (loading / data / error) that templates can render directly.
- Handles auth (JWT access + refresh) via pluggable interceptors.
- Cooperates with the Registry (`registry.provide(apiToken, client)`).

Without this, every consumer manually types `fetch` responses, drifts from backend schema, and re-implements loading/error rendering.

## Motivating use cases from Culinary

1. `GET /v1/ingredients?query=&category=&limit=50` ‚Äî paginated list, live filters, cancel in-flight on new query.
2. `POST /v1/ingredients` ‚Äî form submit; returns created ingredient with server-computed fields.
3. `GET /v1/dish/{id}` ‚Äî single fetch, hydrate detail view.
4. `PUT /v1/dish/{id}/version/{versionId}/graph` ‚Äî bulk graph update payload.
5. `GET /v1/ingredients/similar?name=&nutrients=[...]&category=` ‚Äî debounced live search, must cancel previous.
6. `POST /v1/auth/refresh` ‚Äî silent refresh triggered by 401 response; retry original request once.
7. WebSocket `/v1/dish/{id}/live` ‚Äî future (not MVP, but design must not preclude).

## Proposed API

### Codegen (build-time)

CLI extension:

```
uidetox openapi --input ./openapi.json --output ./src/generated/api.ts
```

Generated file exports:
- Fully-typed request/response types per operation.
- `createClient(baseUrl, opts)` factory returning a client with typed methods grouped by tag: `client.ingredients.list(...)`, `client.dishes.get(...)`.

### Client instance

```ts
import { createClient } from './generated/api';
import { registry, createToken } from 'uidetox';

export const apiToken = createToken<ApiClient>('api');

const api = createClient('/api/v1', {
  auth: {
    getAccessToken: () => tokenStore.access,
    onRefresh: async () => {
      const r = await fetch('/api/v1/auth/refresh', {
        method: 'POST',
        credentials: 'include',
      });
      const { access } = await r.json();
      tokenStore.access = access;
      return access;
    },
  },
  interceptors: [/* optional user hooks */],
});
registry.provide(apiToken, api);
```

### Consumption inside a component ‚Äî three levels

**1. One-shot call (imperative):**

```ts script
const api = registry.get(apiToken).value;
async function saveIngredient(v: Ingredient) {
  await api.ingredients.create({ body: v });
}
```

**2. Reactive resource (declarative):**

```ts script
import { resource } from 'uidetox/http';

const query = state({ text: '', category: null });
const list = resource(
  () => api.ingredients.list({ query: query }),
  { key: () => JSON.stringify(query) }
);
// list.loading, list.data, list.error, list.reload(), list.abort()
```

```html template
<case on=${list.status}>
  <when is="loading"><Spinner/></when>
  <when is="error"><Alert msg=${list.error?.message}/></when>
  <else>
    <for each=${list.data.items} item="ing" key="ing.id">
      <IngredientRow data=${ing}/>
    </for>
  </else>
</case>
```

**3. Mutations with optimistic update:**

```ts
const mutate = mutation(
  (id: string, patch: Partial<Ingredient>) => api.ingredients.update({ path: { id }, body: patch }),
  {
    onOptimistic: (id, patch) => {
      const item = list.data?.items.find(i => i.id === id);
      if (item) Object.assign(item, patch);   // Proxy write triggers UI update
    },
    onRollback: (id, prevSnapshot) => { /* restore */ },
    onSuccess: () => list.reload(),
  }
);
```

### Auto-cancel on component unmount

Every `resource` and every in-flight `client.method()` call originating from a component tree is cancelled when the host component disconnects. Implementation: attach an `AbortController` per component instance and pipe it to `fetch(..., { signal })`.

### 401 handling

On response status 401:
1. If refresh is already in flight, queue the request behind it.
2. Otherwise, call `onRefresh`. On success, retry the original request once with the new token.
3. On second failure, emit an `auth:expired` event through the registry ‚Äî the app decides (redirect to login).

### Error shape

Standardise on:

```ts
type ApiError = {
  status: number;
  code?: string;
  message: string;
  fieldErrors?: Record<string, string[]>;   // maps to fm.setErrors()
  cause?: unknown;
};
```

Form-integration hook: `fm.applyServerErrors(err)` moves `fieldErrors` into the form's error map.

### Query serialization

Support: primitives, arrays (repeat-key style: `?tag=a&tag=b`), objects (dot-notation: `?filter.min=0`), `null` / `undefined` ‚Üí omitted.

### Content types

MVP: JSON only. Multipart / form-urlencoded declared but marked TODO.

## Acceptance criteria

- [ ] `uidetox openapi` CLI produces a self-contained TS file with correct types for GET/POST/PUT/PATCH/DELETE, path params, query params, body, response.
- [ ] Discriminated unions in responses type-narrow correctly (`if (res.kind === 'ok')`).
- [ ] `resource()` re-runs when its declared reactive dependencies change (uses UIDetox `effect`).
- [ ] `resource().abort()` and auto-abort on component unmount both work.
- [ ] JWT refresh flow works: single in-flight refresh, queued retries.
- [ ] `mutation()` optimistic + rollback.
- [ ] Server field errors flow to forms via `fm.applyServerErrors`.
- [ ] Test suite: fake fetch, cancellation, 401 refresh + queue, error shape mapping.

## Open questions

1. Ship as `uidetox/http` (in-tree) or separate `@uidetox/http` package? In-tree preferred for MVP.
2. Fetch impl ‚Äî native or wrapper (`ky`-style)? Native is fine.
3. Client-side caching (SWR-style)? Deferred ‚Äî Culinary MVP does not need. `resource().reload()` is enough.
